📍 漏洞原理

​ 文件包含函数加载的参数没有经过过滤或者严格的定义，可以被用户控制来包含其它的恶意文件，导致执行非预期的代码或者造成文件信息泄露。

🌍 文件包含简介

​ 服务器执行 PHP 文件时，可以通过文件包含函数加载另一个文件中的 PHP 代码，并且当作 PHP 来执行，这会为开发者节省大量时间。比如当向网站添加一张新页面时，仅仅需要修改一下菜单文件，而不是更新所有网页中的链接。

🌍 文件包含函数

其中 include 和 require 的主要区别是，include 在包含过程中如果出现错误，会抛出一个警告，程序正常运行；而 require 函数出现错误的时候，会直接报错并退出程序的执行。

而 include_once() 和 require_once() 这两个函数，与前者的不同之处在于它们只包含一次，适用于在脚本执行期间同一个文件有可能被包含超过一次的情况下，你想确保它只被包含一次来避免函数重定义、变量重新赋值等问题。

📍 分类

  • 本地文件包含 LFI

    • 被包含的文件在服务器本地

  • 远程文件包含 RFI

    • 被包含的文件在第三方服务器，需要开启 php.ini 中的 allow_url_fopen 和 allow_url_include 参数。

📍 PHP-include 文件包含漏洞

[该漏洞无关 PHP 版本]

在 PHP 文件包含漏洞中，如果找不到可以包含的文件，则可以通过包含临时文件的方法来 getshell。如果目标网站上存在 phpinfo，则可以通过 phpinfo 来获取临时文件名，进而进行文件包含。

⚙️ 利用条件：

  • 开启了 php 文件包含函数（存在文件包含漏洞的页面）。
  • 目标文件下有 phpinfo 函数的文件。

⚙️ 利用原理：

在给 PHP 发送 POST 数据包时，如果数据包里包含文件区块，那么无论访问的代码中有没有处理文件上传的逻辑，PHP 都会将这个文件保存成一个临时文件（通常是 /tmp/php[6个随机字符]），文件名可以在 $_FILES 变量中找到，这个临时文件夹通常在请求结束后会被删除。

同时，因为 phpinfo 页面会将当前请求上下文中的所有变量都打印出来，所以如果我们向 phpinfo 页面发送包含文件区块的数据包，那么就可以在返回包中找到 $_FILES 变量的内容，其中自然包含临时文件名。

但是文件包含漏洞和 phpinfo 页面通常是两个页面，理论上需要先发送数据包给 phpinfo 页面，然后从返回页面中匹配出临时文件名，然后再将这个文件名发送给文件包含漏洞页面，进行 getshell。在第一个请求结束时，临时文件就被删除了，第二个请求自然也就无法进行包含。

所以这里利用的方法是：

1）发送包含了 webshell 的上传数据包给 phpinfo 页面，这个数据包的 header、get 等位置需要塞满垃圾数据。

2）因为 phpinfo 页面会将所有数据都打印出来，1）中的垃圾数据会将整个 phpinfo 页面撑的非常大。

3）php 默认的输出缓冲区大小为 4096，可以理解为 php 每次返回 4096 个字节给 socket 连接。

4）因此我们直接操作原生 socket，每次读取 4096 个字节，只要读取到的字符里包含临时文件名，就立刻发送第二个数据包。

5）此时，第一个数据包的 socket 连接实际上还没有结束，因为 php 还在继续每次输出 4096 个字节，所以临时文件此时还没有被删除。

6）利用这个时间差，第二个数据包，也就是文件包含漏洞的利用，即可成功包含临时文件，最终 getshell。

📍 PHP 文件包含漏洞修复方案

1）严格检查变量是否已经初始化。

2）严格判断包含中的参数是否外部可控。

3）基于白名单的包含文件验证，验证被包含的文件是否在白名单中。

4）尽量不要使用动态包含，可以在需要包含的页面固定写好，比如 include(“func.php”)。

5）对所有输入提交可能包含的文件地址，包括服务器本地文件及远程文件，进行严格的检查，参数中不允许出现 ../ 之类的目录跳转符。

6）可以通过调用 str_replace() 函数实现相关敏感字符的过滤，一定程度上可以防御远程文件包含。
